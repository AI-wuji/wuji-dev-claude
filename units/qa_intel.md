# 质监局 SOP

## 验收标准（5项必检）

每个子任务产出必须通过全部5项：

| # | 标准 | 检查内容 |
|---|------|---------|
| 1 | ✅ 功能完整性 | 实现了需求描述的每个点，无遗漏 |
| 2 | ✅ 代码质量 | 简洁、无冗余、命名清晰、无多余注释 |
| 3 | ✅ 安全基线 | 无注入/XSS/硬编码密钥/危险导入 |
| 4 | ✅ 错误DNA预检 | 不引入 ERRORS.md 中已知的错误模式 |
| 5 | ✅ 测试验证 | 修改可被验证，实际跑过不是"应该可以" |

### 不通过处理

```
1项不通过 → 标出具体问题，返回对应师团修复
2项及以上不通过 → 全量返回，附带质监局详细反馈
同一师团连续2次不通过 → 阿极亲自介入，重新评估方案
```

## 错误DNA维护

### 修完bug后（必须做）
在项目根目录 `.claude/ERRORS.md` 追加一条：
```
## N. [类别] 简述
- 涉及文件: xxx
- 现象: xxx
- 根因: xxx
- 修复方法: xxx
- 修复验证: xxx
- 相关文件: xxx
- 错误签名: xxx
- 日期: YYYY-MM-DD
```

### 改代码前（必须做）
- 打开 ERRORS.md，搜索要改的文件名
- 有历史记录的，先读一遍
- 确保这次不会重蹈覆辙

### 去重检测
```
python scripts/errors_db.py dedup "<新错误的签名关键词>"
```
- 相似度 ≥ 70% → ⚠️ 告警，可能重复
- 相似度 ≥ 90% → 🚫 强烈疑似重复，停止检查
- 第 3 次出现同一模式 → 强制根因分析

### 错误DNA数据库命令
```
python scripts/errors_db.py add <类别> <简述> <文件> <现象> <根因> <修复> <验证> <相关文件> <签名>
python scripts/errors_db.py check <文件名>       # 查某文件的历史错误
python scripts/errors_db.py search <关键词>      # 搜索错误
python scripts/errors_db.py dedup <签名>         # 去重检测
python scripts/errors_db.py list                 # 列出全部
```

## 代码审查要点

### 检查清单
```
□ 有没有命令注入？（不拼接shell字符串）
□ 有没有XSS风险？（用户输入要转义）
□ 有没有SQL注入？（用参数化查询）
□ 有没有硬编码密钥？
□ 文件操作是否安全？（不删文件）
□ 依赖是否安全？（不引入已知漏洞的库）
□ 错误处理是否合理？（只在系统边界）
□ 代码是否简洁？（3行能搞定的不写10行）
```

# 第五师（情报作战师）SOP

## 搜索策略 — 多路并行

收到搜索任务后，同一轮同时搜索：

```
并行搜索（WebSearch + WebFetch + Github API）：
├─ 关键词 + "best practice 2026"
├─ 关键词 + "github sort:stars"
├─ 关键词 + "开源方案/库/框架"
├─ 关键词 + "常见问题/坑/避雷"
├─ 竞品/同类产品 分析
└─ 安全/许可证 审查
```

### Claude Code 搜索工具映射

| 搜索目标 | 工具 | 用法 |
|---------|------|------|
| 网页搜索 | WebSearch | 关键词搜索，获取URL列表 |
| GitHub仓库 | `gh search repos` / Github API | `curl -s "https://api.github.com/search/repositories?q=关键词"` |
| 具体网页 | WebFetch | 获取页面内容并提炼 |
| 代码搜索 | Grep / Glob / Agent(Explore) | 在本地代码库搜索 |
| NPM/PyPI | `npm search` / `pip index` | CLI直接查 |
| 安全漏洞 | WebSearch + CVE关键词 | 查已知漏洞 |

## 情报评估 — 五维打分

每个搜索结果标注：

| 维度 | 评分 | 说明 |
|------|------|------|
| 来源 | ⭐⭐⭐ | Star数/社区口碑/作者背景 |
| 可信度 | 高/中/低 | 官方文档>知名博主>论坛帖子>AI生成 |
| 可融合性 | 高/中/低 | 能否直接引用而非复制 |
| 活跃度 | 热/温/冷 | 最近更新时间 |
| 安全评级 | ✅安全 / ⚠️注意 / 🚫危险 | 许可证+已知漏洞 |

## 调研决策 — 五维评估

| 维度 | 评估内容 |
|------|---------|
| 功能匹配 | 是否满足无极的实际需求 |
| 代码质量 | 结构清晰、安全可靠 |
| 活跃度 | 最近更新、社区活跃、issue响应 |
| 可融合性 | 能直接 import 调用，而非复制代码 |
| 协议 | 许可证是否允许商业/闭源使用 |

## 融合原则

```
✅ 融合（引用优于复制）：
  - 已有成熟库 → import 使用
  - 已有稳定模块 → 直接调用
  - 多个方案互补 → 融合精华

❌ 独立开发：
  - 没有类似实现
  - 现有实现质量差/不安全
  - 需要深度定制
```
