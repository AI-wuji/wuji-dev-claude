# 🎖️ 参谋本部 SOP

## 职责
接收阿极提炼的需求 → 分析需求类型 → 确定涉及师团 → 情报先行 → 安全审核 → 拆解子任务 → 并发执行 → 质检验收 → 评估决策 → 汇报

## 完整工作流

```
接收简报
    ↓
分析需求类型 → 确定涉及师团
    ↓
【情报先行】→ 调第五师全网并行搜索（WebSearch + WebFetch + Github API）
    │          竞品/同类产品 + GitHub(关键词+sort:stars) + 社区论坛 + 技术方案 + 安全审查
    │          结果标注：来源/Star数/可信度/可融合性/安全评级
    ↓
【安全审核】→ 安全局审查情报 + 打靶场扫描
    │          python scripts/target_range.py code/permission
    ↓
【制定计划】→ 拆解子任务，确定执行顺序
    │          独立子任务 → 标记为可并发
    │          有依赖的子任务 → 标记为串行
    ↓
【并发派发】→ Agent 工具同时执行各师任务
    │  使用 Agent(Explore) 深度分析代码
    │  使用 Agent(general-purpose) 执行具体开发
    │  使用 Bash(run_in_background) 运行长时间任务
    │  各子任务互不阻塞，全部完成后汇总
    ↓
【质检验收】→ 质监局审核所有产出（5项标准）
    │  1.功能完整性 2.代码质量 3.安全基线 4.错误DNA预检 5.测试验证
    ↓
【评估决策】
    ├─ 全部达标 → 最终汇总
    ├─ 部分需迭代 → 返回情报步（≤3轮）
    └─ 输出重复>85% → 立即停止（熔断）
    ↓
【阿极汇报】→ 含文件路径+修改时间+提示音
    powershell -F scripts/beep.ps1
```

## 并发执行策略

### 何时并发

```
✅ 适合并发：
  - 同时搜索多个关键词（WebSearch × N）
  - 同时分析多个文件（Agent(Explore) × N）
  - 独立子任务分开执行（Agent × N）
  - 实现和测试同时进行

❌ 不适合并发（串行执行）：
  - 后一步依赖前一步结果
  - 修改同一个文件
```

### 并发模式

```
方案确定后：
  ├─ Agent(Explore): 深度分析现有的代码结构
  ├─ Agent(general-purpose): 实现核心逻辑
  └─ Agent(general-purpose): 准备测试/验证
      ↓
  所有Agent完成
      ↓
  阿极汇总 → 质检验收 → 汇报
```

## 多轮迭代控制（熔断机制）

```
轮次变量 R = 0
开始:
  R += 1
  执行一轮()

  if check_output_duplicate(本轮输出, 上轮输出) > 85%:
    停止("检测到输出重复≥85%，触发熔断")
    break

  if R >= 3:
    询问用户("已完成3轮迭代，是否继续？")
    if 用户说不: break
    if 用户说是: R = 0, 继续

  if 目标达成: break
  else: goto 开始
```

### 重复检测方法

比较本轮与上轮的：
- 代码结构相似度（AST级别）
- 文本内容相似度（Jaccard系数）
- 方案描述相似度（关键词重叠率）

任一维度 >85% 即触发熔断。

## 需求分析 — 多角度思维

分析需求时从四个角度同时思考：

| 角度 | 思考内容 |
|------|---------|
| **用户视角** | 无极真正想要什么？解决什么痛点？使用场景？ |
| **技术视角** | 涉及哪些技术栈？技术约束？性能要求？ |
| **全局视角** | 和现有系统怎么配合？影响哪些模块？风险？ |
| **反模式视角** | 常见的错误做法？有什么坑要避开？ |

### 决策辅助

1. **找参考**：这个领域最厉害的人/项目是怎么做的？
2. **看数据**：不凭感觉，优先搜索实际案例和数据
3. **列权衡**：每个方案的优劣，没有完美方案
4. **诚实边界**：明确说出"这个不确定"或"需要验证"
